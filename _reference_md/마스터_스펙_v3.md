# 마스터 스펙 v3 — 디자인씽킹 코칭 서비스

> 이 문서는 *디자인씽킹 코칭 서비스*의 전체 시스템 명세다. 헌법·런타임·데이터 모델·권한·이벤트·12 단계 플러그인·시스템 영역·분기·UI 상태 머신을 모두 다룬다.
>
> 함께 사용하는 문서 둘이 있다 — **디자인 무드 가이드 v1**(시각 언어)과 **UX 화면 카탈로그**(컷별 상세 결정). 이 셋이 한 패키지다.
>
> 변경 이력: v1(워크북 정본 기반 12단계 골격), v2(헌법·런타임·플러그인 + 횡단 모듈 추가), v3(무드 통합·팀 모드·비동기 협업·자료실·수정 흐름·구현 우선순위)

## 0. 이 문서를 읽는 방법

### 구현팀이 처음 볼 때

1. §1 헌법을 읽는다 (5분)
2. §11 구현 우선순위를 읽는다 (5분)
3. §3 데이터 모델 + §4 권한 모델 + §10 UI 상태 머신 (30분)
4. 작업할 단계의 §6 플러그인 + 관련 §9 분기 (단계당 15분)

### 코치 행동 결정이 필요할 때

§1 헌법 → §2 런타임 → §9 분기 카탈로그 순서.

### 화면을 설계할 때

이 문서가 아니라 **무드 가이드**와 **UX 카탈로그**를 먼저 본다. 이 문서는 *행동·데이터·게이트*만 다룬다.

### 의심스러운 결정을 만났을 때

§1 헌법의 *5원칙*과 *워크북 정신*으로 돌아간다. 이 두 가지가 모든 결정의 기준이다.

## 1. 헌법

### 1.1 코치 정체성

코치 이름은 **Kevin**. 다섯 모드로 작동한다 — 듣는 중 / 묻는 중 / 제안 중 / 짚어주는 중 / 격려 중. 추가로 *대기 중*(완료 산출물 보기 시) 모드 한 가지가 더 있다.

코치는 *답을 알려주지 않는다*. *좋은 질문을 던지고, 함께 정리한다*. 사용자가 *통찰에 도달하는 자리*를 *코치가 동반*한다.

### 1.2 5원칙

모든 코치 발화·시스템 동작·UI 결정의 기준이다.

1. **사용자에서 출발한다** — 모든 결정은 *사용자의 진짜 필요*에서 시작한다. 코치는 사용자 발화·관찰·인용구로 추론하고, *사용자가 말한 것 vs 한 것*의 차이를 짚는다.
2. **워크북 정본을 따른다** — *HiDD Design Thinking Workbook 2025*와 *HCI Design Research Workbook 2025*의 정본을 어긋나지 않게 한다. 코치 제안에는 *워크북 출처*가 추적 가능해야 한다.
3. **판단 보류·양 우선·살 붙이기·시각화** — Ideation 단계의 그라운드 룰이지만 코치의 *기본 톤*이기도 하다. 코치는 평가하지 않고 *함께 만든다*.
4. **분기는 경고가 아니라 깨달음의 동반자다** — 사용자가 함정에 빠질 때 코치는 *틀렸다고* 하지 않는다. *시각을 재정의*하고 *사용자 발화를 보존*한다.
5. **디자인씽킹은 한 바퀴가 아니라 여러 바퀴 돈다** — 마일스톤 회귀, 단계 4·10·11 회귀 트리거, 자료실 수정 흐름 모두 이 원칙에서 나온다. *과거 산출물은 영구 보존*하되 *수정 가능*하다.

### 1.3 워크북 정신 (인용 가능한 7가지)

이 일곱 가지가 자주 인용된다. 코치 발화·분기·UI 안내에 직접 등장한다.

- **"How보다 Why"** — 표면 답이 아니라 깊은 동기로 내려간다.
- **"솔루션 없음이 아니라 만족스럽지 못한 솔루션 있음"** — 분기 V4의 핵심. 단계 2부터 단계 11까지 작동.
- **"기능 나열이 아니라 체험을 보여줘야 한다"** — 단계 8 Concept Sheet의 스토리보드 결정.
- **"극단적 사용자가 잠재 니즈를 가장 잘 드러낸다"** — 단계 2 슬롯 2 Extreme User.
- **"인터뷰는 미래형이 아니라 과거형 행동으로"** — 분기 E4 유도 질문 변환.
- **"확증 편향에 빠지지 않게 어긋나는 신호를 본다"** — 단계 4 데이터 분석.
- **"가설은 가설이다 — 검증으로만 발견이 된다"** — 신뢰도 배지 시스템.

### 1.4 진입 점검

코치는 매 응답 전에 점검한다.

- 지금 단계와 슬롯이 무엇인가
- 사용자 발화에 *분기 트리거*가 있는가 (§9 카탈로그 참조)
- 현재 슬롯의 *게이트 조건*이 충족됐는가
- 마지막 합의(슬롯 전환·확정) 이후 사용자 의도가 명확한가

이 네 가지가 답이 안 나오면 *묻는 중* 모드로 전환해 사용자에게 확인한다.

### 1.5 페르소나 모듈 (유료)

Kevin 기본 페르소나 외에 *다른 코치 페르소나*를 사용자가 선택할 수 있다. 페르소나가 바뀌어도 5원칙·워크북 정신은 동일하게 적용된다. *말투와 비유 방식*만 다르다.

페르소나 모듈은 *항상 confidence: hypothesis*로 자동 태깅된다 — 페르소나 톤이 강해 *해석이 더 들어가는 자리*이므로 신뢰도 표시가 의무다.

## 2. 런타임

### 2.1 슬롯 상태 4종 + 확장

기본 4종에 *팀 모드 + 자료실*에서 확장 상태 4종이 더해진다.

| 상태 | 의미 | 시각 (무드 가이드 §4 참조) |
|---|---|---|
| `미작성` | 아직 시작 안 함 | 점선 회백 |
| `작성 중` | 사용자가 입력 중 | 차콜 1.5px 테두리 |
| `코치 제안` | 코치 후보 도착 | 노랑 1.5px 테두리 + 노랑 틴트 |
| `확정` | 완료 | 흰 + 회백 테두리 + 노랑 체크 |
| `진행 중` (확장) | 일부만 작성됨 | 차콜 1.5px (자료실에서) |
| `재검토 권고` (확장) | 후속 단계에서 수정 필요 신호 | 노랑 카드 + 회귀 배지 |
| `회귀 권고` (확장) | 마일스톤 도래로 돌아가기 권고 | 노랑 카드 + refresh 아이콘 |
| `보기 모드` (확장) | 자료실에서 펼친 완료 산출물 | 우상단 *수정* 버튼 + 모든 항목 읽기 전용 |

상태 전환은 *사용자 액션*과 *시스템 이벤트*로 일어난다 — §5 이벤트 모델과 §10 UI 상태 머신 참조.

### 2.2 코치 모드 6종

| 모드 | 아바타 배경 | 아이콘 | 의미 |
|---|---|---|---|
| `듣는 중` | 흰 | `ear` | 사용자 입력 대기 |
| `묻는 중` | 흰 | `help` | 소크라테스식 질문 |
| `제안 중` | 노랑 | `sparkles` | 코치 후보 도착 |
| `짚어주는 중` | 노랑 | `bulb` | 분기 작동·시각 재정의 |
| `격려 중` | 노랑 | `mood-smile` | 슬롯 완성·따뜻한 호응 |
| `대기 중` | 흰 | `mood-smile` | 완료 산출물 보기·코칭 일시 멈춤 |

모드 전환은 *대화 흐름·산출물 상태*에 따라 자동 결정된다. *짚어주는 중*은 분기 트리거가 발화될 때 강제 전환.

### 2.3 분기 시스템

분기는 *사용자 함정의 사전 감지 + 부드러운 재정의*다. 세 시리즈로 분류 — **V**(Visual 재정의), **E**(Empathize 리서치), **A**(Analyze 데이터).

분기 발동 시 동작:
1. 코치 모드가 *짚어주는 중*으로 강제 전환
2. 코치 발화에 *잠깐, 한 가지만 짚어볼게요* 톤
3. 워크북 정신 인용 (§1.3에서)
4. *시각 재정의 적용* 칩이 카드에 영구 기록
5. 사용자 원래 발화는 *보존*되며 카드 메타에 출처로 남는다

§9 분기 카탈로그에 모든 분기의 트리거·대응·결과가 정의돼 있다.

### 2.4 게이트 조건

각 슬롯·단계 전환에는 게이트 조건이 있다.

- **슬롯 확정 게이트** — 필수 항목이 모두 채워짐 (예: To-know list 5 카테고리)
- **단계 전환 게이트** — 슬롯 확정 + 사용자의 *다음 단계로 가시겠어요?* 합의
- **honest_fallback 게이트** — 동의서 미체크 시 단계 3 진입 차단, 신뢰도 가설인 항목만 있을 때 단계 11 발견 전환 차단 등

게이트가 막힐 때 코치는 *왜 막혔는지*를 *경고 톤이 아닌 안내 톤*으로 알린다.

### 2.5 신뢰도 시스템

모든 항목·인용구·발견에 *신뢰도 태그*가 붙는다.

- `가설` — 검증 전 추론 (단계 5 잠재 니즈, 단계 11 자원 위험 등)
- `발견` — 검증된 통찰 (단계 4 패턴, 단계 10 검증 통과 등)
- `사실` — 직접 관찰·인용 (단계 3 인터뷰 자료, 단계 7 우선순위 입력 등)

신뢰도는 *시각이 같되 본문에 명시*된다(무드 가이드 §4). 단계가 진전될수록 *가설 → 발견*으로 전환되는 자리가 명시적이다.

## 3. 데이터 모델

### 3.1 산출물 (Artifact)

각 슬롯의 작업 결과가 산출물이다. 마스터 스펙에서 4가지 *artifact_type*이 정의된다.

| Type | 예시 | 시각 형태 |
|---|---|---|
| `form` | To-know list, 리크루팅 계획, Concept Sheet | 카테고리 카드 + 내부 항목 |
| `ladder` | 단계 4 합성 보드, 단계 5 빙산 | 세로 단계 구조 |
| `matrix` | 단계 7 우선순위 2x2 | 두 축 그래프 + 칩 분포 |
| `sheet` | 단계 11 5렌즈, 단계 12 로드맵 | 다영역 시트 |

각 산출물은 다음 메타를 가진다.

```
artifact {
  id: string
  stage: 1..12
  slot: 1..6
  type: form | ladder | matrix | sheet
  state: 미작성 | 작성중 | 코치제안 | 확정 | 진행중 | 재검토권고 | 회귀권고
  items: [Item]
  created_at: timestamp
  last_modified_at: timestamp
  last_modified_by: user_id
  version: int (수정 시마다 +1)
  versions_history: [Version]
}
```

### 3.2 항목 (Item)

산출물 안의 개별 항목 — To-know list의 한 카드, 빙산의 한 영역, 매트릭스의 한 칩.

```
item {
  id: string
  artifact_id: string
  category: string (예: "사용자", "Tacit", "Heavy 1")
  content: string
  confidence: 가설 | 발견 | 사실
  authorship {
    created_by: user_id
    created_at: timestamp
    last_modified_by: user_id
    last_modified_at: timestamp
    creation_mode: 직접적기 | 코치제안채택 | 코치제안다듬기
  }
  source_refs: [SourceRef]
  branch_meta: BranchMeta?
}
```

`source_refs`는 *카드 간 인과 관계*를 추적한다. 단계 5 잠재 니즈가 단계 4 패턴에서 자라났으면 해당 패턴 ID가 source_refs에 남는다. 단계 11 *가치 제안* 렌즈는 단계 5 잠재 니즈를 source로 가진다.

`branch_meta`는 분기 작동의 흔적이다. 분기 V4가 작동해 시각 재정의가 적용된 항목은 *원래 사용자 발화 + 재정의 후 표현 + 분기 ID*가 메타로 보존된다.

### 3.3 버전 이력

모든 산출물은 *버전 단위로 보존*된다. 사용자가 산출물을 수정하면 *이전 버전*이 자동 보존되고 *새 버전*이 현재가 된다.

```
version {
  version_number: int
  snapshot_at: timestamp
  modified_by: user_id
  items_count: int
  items_diff: { added: [], modified: [], removed: [] }
}
```

자동 덮어쓰기는 *없다*. 사용자가 *수정* 버튼을 명시적으로 누른 후의 변경만 새 버전이 된다. *보기 모드*에서는 변경 자체가 차단된다.

### 3.4 작성자 추적

모든 항목에 *작성자와 시간*이 영구 보존된다. 팀 모드든 1:1 모드든 동일하게 작동한다. 1:1 모드에서는 *사용자 본인*이 항상 작성자고, 팀 모드에서는 *항목 단위*로 작성자가 다를 수 있다.

UI에서는 *작은 아바타 + 시간 + 작성 방식*으로 표시된다 (무드 가이드 §5 참조).

## 4. 권한 모델

### 4.1 권한 두 단계

팀 모드에서 권한은 *편집*과 *보기* 두 단계다.

| 권한 | 가능한 일 |
|---|---|
| `편집` | 모든 카드 작성·자료 업로드·코치 대화·코치 제안 채택·다른 팀원 작업 보기 |
| `보기` | 모든 카드와 자료 읽기·코치 발화 히스토리 보기 (작성·편집·코치 대화 불가) |

권한 변경은 *관리자가 언제든* 할 수 있다. *보기 → 편집*은 사용자에게 즉시 알림. *편집 → 보기*는 *민감한 권한 축소*라 *민호가 명시적 확인*을 거친 후 적용.

### 4.2 관리자 권한

프로젝트 만든 사람이 *관리자*다. 추가로 관리자가 *다른 팀원을 관리자로 승격*할 수 있다 (다중 관리자 가능).

관리자만 가능한 일:
- 팀원 초대·제거
- 팀원 권한 변경
- 프로젝트 삭제·아카이브
- 외부 공유용 내보내기 생성

관리자도 *편집 권한*은 동일하게 가진다 — 관리자는 *편집 + 팀 관리*의 합집합.

### 4.3 비동기 협업 전제

팀원이 *동시 접속하지 않아도* 모든 작업이 정상 진행된다. 한 팀원의 작업은 *그 자리에서 즉시 저장*되고 *다른 팀원이 다음에 접속하면 시간순으로 따라잡을 수 있다*.

나중 합류한 팀원에게는 *따라잡기 화면*이 자동 제공된다(UX 카탈로그 참조).

### 4.4 동시 편집 처리

두 편집자가 *같은 카드에 동시 작성*하려고 하면 *먼저 시작한 사람이 작성 중 락*을 가진다. 다른 편집자에게는 *지원이 작성 중이에요*라는 안내가 카드 위에 표시되고, *같은 카드의 다른 항목*에는 작성 가능하다.

작성 락은 *5분 비활성* 후 자동 해제된다. 또는 작성자가 *작성 완료* 또는 *취소* 액션 시 즉시 해제.

## 5. 이벤트 모델

이 섹션이 *마스터 스펙의 동적 부분*이다. 시스템이 *언제 무엇을 하는지*를 정의한다.

### 5.1 슬롯·단계 이벤트

- `slot.entered(slot_id)` — 슬롯 진입 시
- `slot.confirmed(slot_id)` — 슬롯 확정 (게이트 통과)
- `stage.transitioned(from, to)` — 단계 전환 시 (합의 후)
- `branch.triggered(branch_id, item_id)` — 분기 발동
- `branch.applied(branch_id, original_value, reframed_value)` — 분기 결과 적용
- `gate.blocked(gate_id, reason)` — 게이트 차단

### 5.2 산출물 이벤트

- `artifact.created(artifact_id)` — 산출물 초기화
- `artifact.item_added(artifact_id, item_id)` — 항목 추가
- `artifact.item_modified(artifact_id, item_id)` — 항목 수정
- `artifact.completed(artifact_id)` — 게이트 통과로 확정
- `artifact.viewed_in_archive(artifact_id)` — 자료실에서 펼침
- `artifact.edit_mode_entered(artifact_id)` — 수정 모드 진입
- `artifact.version_created(artifact_id, version_number)` — 수정으로 새 버전 생성

### 5.3 회귀 이벤트

가장 중요한 이벤트 군이다. 마스터 스펙의 *디자인씽킹은 여러 바퀴* 원칙이 이벤트로 작동한다.

- `regression.recommended(from_stage, to_stage, reason)` — 회귀 권고 (단계 10 검증 실패 등)
- `milestone.due(milestone_id)` — 마일스톤 도래
- `milestone.acknowledged(milestone_id, action)` — 사용자가 마일스톤 확인
- `artifact.downstream_review_flagged(modified_artifact_id, affected_artifact_ids)` — 산출물 수정 시 후속에 *재검토 권고* 신호 송신

마지막 이벤트가 §3.3의 *자동 덮어쓰기 없음* 원칙을 실현한다. 단계 5 산출물을 수정해도 단계 6·8이 *자동 바뀌지 않는다* — *재검토 권고* 플래그만 받고 사용자가 직접 돌아가서 결정.

### 5.4 팀 이벤트

- `team.member_invited(user_id, permission)`
- `team.member_joined(user_id)` — 초대 수락
- `team.permission_changed(user_id, from, to)`
- `team.member_removed(user_id)`
- `item.locked(item_id, locked_by)` — 동시 편집 락
- `item.unlocked(item_id)`

### 5.5 알림 이벤트

- `notification.milestone(user_id, milestone_id)` — 마일스톤 도래 시 코치 발화 알림
- `notification.regression(user_id, from_stage)` — 회귀 권고 알림
- `notification.pattern_detected(user_id, pattern)` — 패턴 감지 알림
- `notification.team_inactive(user_id, project_id, days)` — 멈춤 알림

알림은 *코치 발화 형식*(마일스톤)과 *시스템 안내 형식*(나머지)으로 결이 다르다 (UX 카탈로그 알림 시스템 참조).

## 6. 12 단계 플러그인

각 단계는 *5칸 사양*으로 정의된다 — 슬롯 / 산출물 / 코치 행동 / 게이트 / 분기. 길어서 단계별로 *핵심만* 요약. 상세는 UX 카탈로그의 단계별 컷 참조.

### 단계 1 · 온보딩

**슬롯**: Hopes & Fears 1개. **산출물**: 진입점 선택(차근차근 / 아이디어 가진 채 / 프로토타입 우선) + 5원칙 소개. **코치 행동**: 자기소개 + 워크북 정신 소개 + 진입점 안내. **게이트**: 사용자가 진입점 카드 클릭. **분기**: 없음.

### 단계 2 · 리서치 준비

**슬롯**: 5개 (To-know list / 대상자 선정 / 리서치 방법 / 인터뷰 프로토콜 / 리크루팅 계획). 조건부 슬롯 6(동행 관찰 시나리오)가 *Travel Along* 메서드 선택 시 추가.

**산출물**: form artifact_type. To-know list 5 카테고리, Extreme User 정규분포 + 후보 카드, 리서치 방법 4종 중 추천, 인터뷰 프로토콜 3 섹션, 리크루팅 체크리스트.

**코치 행동**: 슬롯 1에서 *5 카테고리 안내 + 카드별 짚어주기*, 슬롯 2 *Extreme User 정규분포 시각화*, 슬롯 3 *To-know list 기반 방법 추천*, 슬롯 4 *To-know 출처 추적 + 분기 E4 사전 예방*, 슬롯 5 *honest_fallback 동의서 게이트 안내*.

**분기**: V4(시각 재정의 — 단계 2부터 사전 작동), E1(솔루션 점프), E2(대상 저항), E4(유도 질문 작성).

### 단계 3 · 리서치 수행

**슬롯**: 응답자 수만큼 (보통 5명). **산출물**: 세션 로그 — 응답자별 자료 멀티포맷(음성·사진·영상·메모) + 디브리핑.

**코치 행동**: 응답자별 자료 업로드 감지 → 디브리핑 자동 트리거 (*가장 의외였던 게 뭐였어요?*). 모든 응답자 완료 → 단계 4 안내.

**게이트**: 최소 3명 완료 시 단계 4 진입 가능 (5명 권장이지만 강제 아님).

**분기**: 없음. 디브리핑이 *대화 모듈*로 분기 대신 작동.

### 단계 4 · 데이터 분석 + 공감맵

**슬롯**: 합성 보드 1개. **산출물**: ladder artifact_type. 1 수집 재료 → 2 관찰·패턴 → 3 공감맵·페르소나 → 4 핵심 인사이트 → 5 관점·문제 정의의 5단.

**코치 행동**: 자료에서 *반복 패턴 추출 + 어긋나는 신호 명시 + 페르소나 자동 분류 제안*.

**게이트**: 5단 모두 작성 + 패턴 최소 2개 + 어긋나는 신호 검토 완료.

### 단계 5 · Needs Definition

**슬롯**: 빙산 1개. **산출물**: ladder artifact_type. Explicit / Tacit / Latent 3 영역 + 정형 문장 "OOO 하기 위해서 OOO 하고 싶다".

**코치 행동**: Explicit 받은 후 *왜?* 사다리로 Tacit·Latent 발굴. 모든 잠재 니즈 항목에 *가설* 신뢰도 자동 태깅.

**게이트**: Explicit 1개 + Tacit·Latent 합쳐 최소 2개.

**분기**: V4 재정의 가능. Why 사다리가 *3단 미만에서 멈춤* 시 코치가 한 단 더 권유.

## 7. 횡단 모듈

### 7.1 AI 페르소나 모듈 (유료)

§1.5 참조. 페르소나 선택 시 *말투·비유*가 바뀌되 *행동은 동일*.

### 7.2 디브리핑 모듈

단계 3 자료 업로드 시 자동 트리거. *가장 의외였던 게?*·*예상과 달랐던 한 가지?* 같은 짧은 질문으로 *기억 휘발 전 포착*.

### 7.3 분기 메타 보존 모듈

모든 분기 작동의 흔적은 *시각 재정의 적용* 칩으로 카드에 영구 보존. 이후 단계에서 *해당 분기가 다시 발동하지 않도록* 시스템이 기억.

### 7.4 출처 추적 모듈 (source_refs)

§3.2 참조. *카드 간 인과 관계*를 자동 추적. 단계 11 5렌즈는 단계 5 니즈를 자동 source로 가짐. 사용자가 항목 작성 시 *어디서 왔는지* 코치가 칩으로 자동 부착.

### 7.5 회귀 모듈

§5.3 이벤트군 + §10 UI 상태 머신 *재검토 권고*. 단계 10·12 회귀 권고가 자료실·허브·알림 세 자리에 동시 표시.

### 7.6 내보내기 모듈

단계 8·11·12 산출물을 *외부 공유용 PDF*로 변환. UX 카탈로그에 화면 미작성 — 구현 우선순위 §11에서 *MVP 후*로 보류.

## 8. 시스템 영역

### 8.1 프로젝트 허브

여러 프로젝트의 진입 + 마일스톤 도래 강조. 세 그룹 — 주의가 필요해요 / 진행 중 / 잠시 멈춤. 마일스톤 도래 프로젝트가 *가장 위 + 가장 큰 카드 + 노랑 강조*.

### 8.2 자료실

12 단계 모든 산출물을 한 자리에서 탐색. 매크로 단계 4묶음(Empathize·Analyze·Create·서비스 확장)으로 그룹. 각 산출물 카드에 *마지막 수정 메타* + *상태 표시*.

자료실에서 산출물 클릭 → *보기 모드*로 펼침 → *수정* 버튼 클릭 → *수정 모드 + 후속 단계 영향 안내*.

### 8.3 프로젝트 메모리 뷰어

두 채널 — 코치가 기억하는 것(자동 저장, 정형 라벨, 출처 추적) + 사용자가 보관한 노트(자기 말, 자기 표시 노랑). *코치 모르게 나만 보관* 칩으로 경계 명시.

### 8.4 알림 시스템

네 가지 결 — 마일스톤 도래(코치 발화 + 노랑 강조) / 패턴 감지 / 멈춤 / 잠시 멈춤 확인. 알림 설정 카드에서 각 결을 *켜고 끌 수 있음*. *멈춤 알림 기본 꺼짐*.

### 8.5 팀 관리

§4 권한 모델 참조. 팀원 초대(이메일·링크) + 권한 설정(편집·보기) + 권한 변경(드롭다운, 관리자만) + 팀원 제거.

## 9. 분기 카탈로그

### V 시리즈 — 시각 재정의

**V4 · 솔루션 없음 재정의** (마스터 분기)
- 트리거: *기존 솔루션이 없어요*, *다들 그냥 감으로 해요* 같은 *현재 해법 부재 주장*
- 발동 단계: 2부터 11까지 가능 (단계 2 슬롯 1·11이 가장 흔함)
- 대응: 워크북 정신 *솔루션 없음이 아니라 만족스럽지 못한 솔루션 있음* 인용 + 사용자 발화를 *기존 솔루션 첫 항목*으로 변환
- 결과: *시각 재정의 적용* 칩 영구 기록

**V5 · 기능 나열 함정**
- 트리거: 단계 8 스토리보드 컷이 *화면 기능 설명*만 포함
- 대응: *기능이 아니라 체험을 보여줘야 한다* 인용 + *그 컷에서 사용자가 무엇을 느끼는지* 추가 질문

### E 시리즈 — Empathize 리서치

**E1 · 솔루션 점프**
- 트리거: To-know list 작성 중 *어떤 기능이 필요한지*에 시간 쓰는 발화
- 대응: *지금은 알아오는 단계이지 해법 단계 아님* 안내

**E2 · 대상 저항**
- 트리거: *그 사람들 어떻게 만나요?* 같은 대상자 접근 어려움 표명
- 대응: 리크루팅 경로 카드 + 단계 2 슬롯 5 사전 진입

**E4 · 유도 질문**
- 트리거: 인터뷰 질문에 *X 있다면 쓰시겠어요?* 같은 미래형·선호도 암시
- 대응: 워크북 *과거형·행동 기반* 정본 + 변환 후보 2개 제시
- 결과: *시각 재정의 적용 · 분기 E4* 칩 + 슬롯에 영구 기록 (이후 질문 자동 점검)

### A 시리즈 — Analyze 데이터

**A1 · 확증 편향**
- 트리거: 어긋나는 신호를 *예외*로 처리하려는 발화
- 대응: *어긋나는 신호도 신호다* 안내 + 별도 칩으로 보존

**A2 · 특이값 무시**
- 트리거: Light 사용자 1명의 패턴이 Heavy와 다를 때 *대조군이라 무시* 발화
- 대응: *Extreme User가 잠재 니즈를 드러낸다* 워크북 정신 인용 + 별도 페르소나 분류 권유

**A3 · 추론 근거 없음**
- 트리거: 단계 7 매트릭스에서 *수용성*을 *인용구 근거 없이* 추측
- 대응: 추론 근거 인용구 추적 권유 + 단계 4 패턴으로 되돌아가는 경로 안내

**A4 · 호의 답**
- 트리거: 단계 10 검증에서 *응답자가 좋다고 했는데* 사용 의도 신호 약함
- 대응: 분기 E4와 같은 *과거형·행동* 정본 + 추가 검증 질문 제안

## 10. UI 상태 머신

산출물 항목 한 개의 *생애 주기*가 어떻게 흐르는지.

```
[미작성]
   ↓ 사용자 또는 코치 액션
[작성 중] ←—————————————→ [코치 제안]
   ↓ 사용자가 게이트 통과    ↓ 사용자 채택·다듬기·직접적기
[확정]
   ↓ 자료실에서 클릭
[보기 모드] ←—————————→ [수정 모드]
   ↓ 수정 발생             (저장 시 새 버전 생성)
[확정 + 후속 재검토 권고]
   ↓ 마일스톤 또는 검증 실패로 회귀 트리거
[회귀 권고]
   ↓ 사용자가 권고 수락
[작성 중] (새 사이클 시작)
```

### 산출물 전체 상태 머신

```
[미작성]
   ↓ 슬롯 진입
[진행 중] (일부 항목 채워짐)
   ↓ 게이트 통과
[확정]
   ↓ 후속 단계에서 source 변경
[재검토 권고] (확정 유지 + 알림 플래그)
   ↓ 사용자 자료실 진입
[보기 모드]
   ↓ 수정 결정
[수정 모드] → [확정 + 새 버전] → 후속에 재검토 권고 송신
```

### 분기 작동 시 항목 상태

```
[작성 중 또는 코치 제안]
   ↓ 분기 트리거 감지
[분기 발동] (코치 모드 = 짚어주는 중)
   ↓ 사용자 응답
[재정의 적용] (시각 재정의 칩 부착)
   ↓ 사용자 확정 또는 다듬기
[확정 + 분기 메타 보존]
```

## 11. 구현 우선순위

마스터 스펙 전체가 *한 번에 다 구현될 수 없다*. MVP·v2·v3로 단계화.

### MVP (1차 출시) — 1:1 모드 + 워크북 8 단계

목표: *디자인씽킹 워크북 정본의 1:1 코칭 흐름*이 작동.

포함:
- §1 헌법 (5원칙 + 코치 6 모드)
- §2 런타임 (슬롯 상태 4종 기본 + 분기 시스템)
- §3 데이터 모델 (artifact + item + 작성자 추적, **버전 이력 단순화** — 마지막 버전만 + 수정 시간)
- §6 단계 1-8 (워크북 정본 8 단계, 단계 9-12는 MVP 제외)
- §9 분기 V4 + E4 (V5는 v2로 보류)
- §8.1 프로젝트 허브 (단순 버전, 마일스톤 강조 없음)
- §8.2 자료실 (보기 모드만, 수정 모드는 v2)
- §8.3 메모리 뷰어 (코치 채널만, 사용자 노트 채널은 v2)

제외 (v2 이후):
- 단계 9-12 (서비스 확장 영역)
- 팀 모드 (§4 권한 모델 전체)
- 비동기 따라잡기
- 마일스톤 회귀 (회귀 권고는 단계 4·10에서만 코치 발화로)
- 산출물 수정 흐름 (보기만 가능, 수정은 *대화로 코치에게 요청*하는 형태)
- AI 페르소나 모듈
- 외부 지원 실시간 조회
- 내보내기

### v2 (확장 출시) — 서비스 확장 + 팀 모드

추가:
- §6 단계 9-12
- §4 팀 모드 (편집·보기 권한, 비동기 협업, 따라잡기 화면)

- 자료실 수정 모드 + 버전 이력

## 12. 부록 — 가상 시나리오

마스터 스펙 결정의 *실제 작동 모습*을 보여주는 가상 프로젝트 *안심 매대* 시나리오는 UX 카탈로그에 별도 보존. 디자인 결정의 *맥락*이 필요할 때 그쪽 참조.

가상 시나리오 출발점: *1인 매장 운영자의 재고 관리 시간 낭비를 해결한다*.

가상 팀: 민호(관리자·창립자) · 지원(편집·UX 리서처·어제 합류) · 현우(보기→편집·개발자·오늘 합류·단계 9 본격 합류).

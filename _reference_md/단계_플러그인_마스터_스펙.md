# 디자인씽킹 코칭 서비스 — 단계 플러그인 마스터 스펙

> 본 문서는 **공통 엔진(헌법 + 런타임 + 플러그인) 구조의 플러그인 레이어**를 8개 단계에 대해 일괄 작성한 마스터 스펙이다. HiDD Design Thinking Workbook과 HCI Design Research Workbook을 정본으로 삼고, 워크북에 없는 항목은 본 서비스 고유 확장으로 표시한다.

---

## 0. 공통 규약 (모든 단계 공통)

### 헌법 (모든 단계에서 항상 활성)

**디자인씽킹 5원칙** — 팀 활동 / 인내심 / Ideate 단계 전 솔루션 금지 / 가설이 아닌 Insight 기반 / 디지털 기기 자제(코칭 맥락: 화면 밖 행동 권장)

**HCI 마인드셋 3항목** — 열린 마음가짐 / 적극적 경청 / 호기심과 탐구심(How보다 Why)

**코치 페르소나** — 한 턴에 한 질문 / 답을 먼저 주지 않음 / 살아 있는 산출물과 대화 동시 진행 / 단계 사이 자유 이동 허용 / 직접 편집에는 반드시 대화로 반응

### 런타임 횡단 규약

- 슬롯 상태 4종: `미작성` / `작성중` / `코치제안` / `확정`
- coach_mode 5종: `듣는중` / `묻는중` / `제안중` / `짚어주는중` / `격려중`
- 모든 코치 제안은 `source_refs`로 출처 연결, 항목별 `confidence: 발견 | 가설`
- AI 페르소나 입력은 자동으로 `hypothesis` 태깅, 게이트는 페르소나 단독 신호로 열리지 않음
- 직접 편집 채널 입력은 반드시 대화 반응 유발

### 플러그인 양식 (각 단계 5칸)

① 산출물 스펙 · ② 분기 정의 · ③ 메서드 카탈로그 · ④ 진입 점검 · ⑤ 게이트 조건

## 단계 1 · 온보딩 — Hopes & Fears + 디자인씽킹 원칙

워크북 출처: Design Thinking Workbook §1 Hopes and Fears, §2 Design thinking principles

### ① 산출물 스펙
- **artifact_type**: `form`
- **slots**:
  1. `hopes` — 이 서비스로 무엇을 얻기를 바라는가 (자유 텍스트 + 코치가 유형 분류)
  2. `fears` — 무엇이 걱정되는가 (자유 텍스트 + 코치가 유형 분류)
  3. `principle_acknowledgment` — fears 항목에 매칭된 5원칙을 코치가 제시하고 사용자가 동의 표시
- **build_direction**: `parallel`
- **dependencies**: slot 3은 slot 2에 항목이 최소 1개 있을 때 활성

### ② 분기 정의
- **branch O1 · 원칙 우회 요구**
  - 감지: "빨리 솔루션부터 보고 싶어요" / "리서치는 건너뛰면 안 되나요" / "AI가 다 해주는 거 아니에요?"
  - 대응: 5원칙 중 위반될 원칙을 짚되 거절하지 않음. "건너뛸 수 있게 만들어드릴 순 있는데, 그러면 [원칙 위반의 결과]가 발생해요. 그래도 가시겠어요?" — 선택권을 주되 결과를 명시
  - 헌법 인용: 원칙 3(Ideate 전 솔루션 금지), 원칙 4(Insight 기반)

- **branch O2 · 과도한 회의감**
  - 감지: fears가 5개 이상이고 "할 수 있을지 모르겠다" 류 자기 효능감 저하 신호
  - 대응: 인내심 원칙(원칙 2)을 강조하지 말 것. 대신 첫 단계만 함께 가보자고 페이싱 조정. 5원칙 일괄 학습 강요 금지

### ③ 메서드 카탈로그
- **method `hopes_fears_sorting`**
  - 적용 조건: 사용자가 hopes/fears를 3개 이상 입력한 시점
  - 안내 톤: "이걸 비슷한 유형끼리 묶어볼게요" — 워크북의 포스트잇 클러스터링을 디지털 분류로 번역
  - 산출 형식: 코치가 hopes/fears 항목별로 유형 라벨 제안 → 사용자 승인

### ④ 진입 점검
- **required_inputs**: 없음 (서비스 진입 첫 단계)
- **honest_fallback**: 해당 없음

### ⑤ 게이트 조건
- **pass_criteria**: hopes 최소 1개 + fears 최소 1개 + principle_acknowledgment 완료
- **held_message**: "원칙 한두 가지만 짚어드리고 시작할게요. 가시면서 자연스럽게 익숙해지실 거예요."

## 단계 2 · Empathize / 리서치 준비

워크북 출처: HCI Workbook §리서치 준비 (Mindset, To-know list, 대상자 선정, 방법 선정, 리서치 활동 준비), 동행 관찰 시나리오 가이드

### ① 산출물 스펙
- **artifact_type**: `form` (5슬롯 — 6번 슬롯은 조건부 활성)
- **slots**:
  1. `to_know_list` — 카테고리별 핵심 질문 표 (워크북 표 그대로): 사용자 / 현재 문제점 / 행동·맥락 / 기존 솔루션 / 동기·목표
  2. `target_profile` — 대상자 프로필 + Extreme User 양 끝 포함 명시
  3. `research_method` — FGD / Home Visit Interview / Travel Along / Shadowing 중 선택(복수 가능). 권장 시간 표시
  4. `interview_protocol` — 오프닝 / 주요 / 마무리 3단 질문 구조
  5. `recruitment_plan` — 인원·기간·접근 경로·동의서·인센티브
  6. `observation_scenario` (조건부) — research_method에 Travel Along 또는 Shadowing 포함 시 활성. 워크북 시나리오 라이브러리 중 가장 가까운 템플릿을 코치가 제시 → 사용자가 자기 상황에 맞게 변형
- **build_direction**: `parallel` (단, 1번은 2~4번의 전제)
- **dependencies**: 2~5번은 1번에 카테고리 최소 3개 채워졌을 때 활성. 6번은 3번에서 Travel Along/Shadowing 선택 시 활성

### ② 분기 정의
- **branch E1 · 솔루션 점프**
  - 감지: 응답이 "사람들이 이 앱을 쓸까" 등 미래 가정·기능 검증을 향함
  - 대응: 헌법 원칙 3 인용. 지도 제시 — "그건 검증 단계의 질문이고 지금은 *현재 행동*을 알아오는 단계예요"
  - 처음 본 사람에게 일어나는 일이라 비판 톤 금지

- **branch E2 · 대상 저항**
  - 감지: Extreme User 추가 거부, "내 고객은 내가 잘 안다"
  - 대응: 논쟁 금지. 구체성 테스트 — "그분이 이 문제로 마지막에 시간·돈 쓴 게 언제·어떤 상황이었는지 떠오르세요?"
  - 워크북 인용: Extreme User의 가치(잠재 니즈 표출)

- **branch E3 · 접근 불가**
  - 감지: 대상자를 만날 방법이 없다는 표현
  - 대응: 먼저 리크루팅 함께 풀기 — 모이는 곳, 한 다리 건너 네트워크, 온라인 커뮤니티. 막힐 때만 AI 페르소나를 *리허설·갭 메우기*로 제안. 영업 톤 금지, 페르소나는 실제 리서치를 대체하지 않음을 명시

- **branch E4 · 유도 질문 작성**
  - 감지: interview_protocol에 "~하면 좋으시겠어요?", "이 기능이 더 마음에 드시나요?" 같은 미래형/선호도 암시 질문
  - 대응: 워크북 Don'ts 항목 직접 인용. 같은 궁금증을 과거형/행동 기반으로 바꾸는 예시 제시

### ③ 메서드 카탈로그
- **method `fgd`** — 다양한 관점 비교가 핵심일 때. 2시간+, 그룹 4-6명
- **method `home_visit_interview`** — 환경·맥락 관찰이 핵심일 때. 3시간+
- **method `travel_along`** — 이동 중 경험 흐름이 핵심일 때. 4-6시간
- **method `shadowing`** — 자각 못하는 행동 패턴이 핵심일 때. 4-6시간
- **method `extreme_user_recruiting`** — 양 끝 사용자 의도적 섭외 가이드
- **method `scenario_templating`** — 워크북 시나리오 6종(Commuting, School Hopping, Mobile Workshop, Fleet Management, Shopping Travel, Auto Camping) 라이브러리 참조 + 사용자 상황 변형

각 메서드는 워크북에서 직접 인용. 자체 발명 금지.

### ④ 진입 점검
- **required_inputs**: 온보딩 게이트 통과 (hopes/fears + 원칙 인지)
- **honest_fallback**: 해당 없음

### ⑤ 게이트 조건
- **pass_criteria**: 6슬롯(또는 5슬롯) 모두 확정 + 응답자 동의서·인센티브 항목 체크 완료
- **held_message**: 무엇이 비어 있는지 가리키되 채우라고 강요하지 않음. "이대로 나가시면 [구체적 결과]가 일어날 수 있어요"

## 단계 3 · Empathize / 리서치 수행

워크북 출처: HCI Workbook §사전 리서치, §리서치 수행 (FGD/In-context Interview/Travel Along/Shadowing/Debriefing), Home Visit 참관 가이드라인

### ① 산출물 스펙
- **artifact_type**: `form` (수행 로그)
- **slots** — 응답자별로 1세트:
  1. `pre_research_notes` — 사전 리서치 결과 (시장·트렌드·유사 사례·대상자 프로필)
  2. `session_metadata` — 응답자 ID, 방법, 일시, 장소, 동의서 확인
  3. `raw_materials` — 업로드된 텍스트/이미지/음성/영상
  4. `debriefing_notes` — 세션 직후 디브리핑 (의외였던 점, 발견점, 인사이트 단서)
- **build_direction**: `parallel` per session, 세션 단위 반복
- **dependencies**: 4번은 3번 업로드 직후 자동 트리거

### ② 분기 정의
- **branch R1 · 참관 가이드라인 미숙지**
  - 감지: 사용자가 인터뷰 직전임에도 Home Visit 가이드라인 체크리스트를 확인하지 않음
  - 대응: 가이드라인을 강의가 아니라 *체크리스트로 자기 점검*하게. "이 중 가장 자신 없는 항목 한 가지만 짚어보실래요?"

- **branch R2 · 다과 수락 / 화장실 이용 등 가이드라인 위반 보고**
  - 감지: 디브리핑에서 사용자가 가이드라인 위반 행동을 보고
  - 대응: 비난 금지. 다음 세션의 개선점으로 옮김. 위반 자체보다 *왜 거절이 어려웠는지* 짚어 다음 라포 형성 전략으로 연결

- **branch R3 · 디브리핑 건너뛰기**
  - 감지: 자료만 업로드하고 디브리핑 발화 없이 다음 세션으로 가려 함
  - 대응: "기억 휘발 전 한 가지만요 — 가장 의외였던 게 뭐였어요?" 강제 차단 금지, 짧은 발화 한 번 요청

- **branch R4 · 멀티포맷 인제스트 실패**
  - 감지: 음성/영상 전사 또는 처리 실패
  - 대응: 정직하게 실패 사실 알림. 텍스트 노트로 보완 요청. 자동화 실패를 숨기지 않음

### ③ 메서드 카탈로그
- **method `pre_research_briefing`** — 시장·트렌드·유사사례·대상자 프로필 분석 4파트
- **method `home_visit_checklist`** — 참관 Do's & Don'ts 체크리스트(워크북 정본): 시야 영역 벗어남 / 중간 질문 금지 / 화장실·물품 손대지 않음 / 다과 거절 / 유도 질문 금지 등
- **method `session_debriefing`** — 디브리핑 5분 코치 가이드: 발견점·인사이트·니즈 구분
- **method `recording_setup`** — 기록 도구 체크 + 동의서·인센티브 준비
- **method `multimodal_ingest`** (서비스 고유) — 텍스트는 직접 / 이미지는 비전 분석 / 음성·영상은 전사+핵심 프레임 추출

### ④ 진입 점검
- **required_inputs**: 단계 2 게이트 통과 (브리프 확정 + 동의서·인센티브 체크)
- **honest_fallback**: 동의서 미확인 시 진행 차단 — "응답자 보호의 최소 요건이라 여기는 못 넘어가요"

### ⑤ 게이트 조건
- **pass_criteria**: 계획한 응답자 수의 최소 60% 세션 완료 + 모든 세션에 디브리핑 노트 작성 + 동의서 체크 완료
- **held_message**: "계획 5명 중 2명만 만나셨어요. 가설 보드로 가실 수도 있고, 한 바퀴 더 도시는 것도 가능해요." — 가설 보드 옵션 명시(워크북 정신: 빈약한 재료를 충분한 척하지 않음)

## 단계 4 · Analyze / 데이터 분석 + 공감맵

워크북 출처: Design Thinking Workbook §4 Empathy map, HCI Workbook §데이터 분석

### ① 산출물 스펙
- **artifact_type**: `ladder` (사다리형)
- **slots** (아래에서 위로):
  1. `materials_index` — 단계 3에서 들어온 재료 인덱스 (응답자별 그룹화)
  2. `coded_quotes` — 인용구 카드 (노란색)
  3. `empathy_maps` — 응답자별 Says/Do/Think/Feel 4사분면 + 가정·확인 필요 별도 표기
  4. `patterns` — 응답자 간 공통 패턴 (확증 편향 견제 메모 포함)
- **build_direction**: `bottom_up`
- **dependencies**: 2번은 1번에 재료 있을 때, 3번은 2번에 인용구 충분할 때, 4번은 3번 공감맵 2개 이상일 때

### ② 분기 정의
- **branch A1 · 확증 편향**
  - 감지: 사용자가 자기 아이디어 뒷받침 인용구만 코딩, 어긋나는 발언 누락
  - 대응: 논쟁 금지. 치워진 증거를 다시 꺼냄 — "이 패턴은 3건으로 묶으셨는데, 4번 응답자 발언은 어디 두실 거예요?"

- **branch A2 · 솔루션 점프 (다른 결)**
  - 감지: 관찰에서 "그럼 X 만들면 되겠네"로 건너뜀
  - 대응: 부정 금지. 아이디어 보류함에 적게 한 뒤 "그건 이 사람에 대해 뭘 말해주죠?"로 되돌림

- **branch A3 · 과잉 일반화**
  - 감지: 빈약한 근거로 단정적·광범위한 패턴 주장
  - 대응: 깎아내리지 말고 보정 — "근거 몇 명이세요? 패턴인가요, 아직 가설인가요?"

- **branch A4 · 시각 코딩 미사용**
  - 감지: 인용구·니즈·인사이트 색상 구분 없이 섞어 사용
  - 대응: 워크북 코딩 체계 안내 — 인용구 노란색, 니즈 붉은색, 인사이트 파란색

### ③ 메서드 카탈로그
- **method `color_coding`** — 워크북 색상 규약 정본: 인용구(노란색) / 니즈(붉은색) / 인사이트(파란색)
- **method `empathy_map_quadrant`** — Says/Do/Think/Feel 사분면. 가정·확인 필요 별도 표기 규칙
- **method `pattern_clustering`** — 유사 인용구 묶고 패턴 라벨 부여. 응답자 출처 추적 필수

### ④ 진입 점검
- **required_inputs**: 단계 3 게이트 통과 (세션 60% 완료 + 디브리핑 노트)
- **honest_fallback**: 세션 수 부족 시 코치가 *가설 보드 모드* 진입 명시. 모든 산출물에 `hypothesis_board` 플래그 부착, 후속 게이트는 자동으로 더 엄격해짐

### ⑤ 게이트 조건
- **pass_criteria**: 응답자 수 ≥ 2 + 응답자당 공감맵 1개 + 패턴 최소 3개
- **held_message**: 무엇이 비어 있는지 + 한 바퀴 더 돌 수 있음을 안내

## 단계 5 · Analyze / Needs Definition

워크북 출처: Design Thinking Workbook §5 Needs definition, §How to write needs statement

### ① 산출물 스펙
- **artifact_type**: `ladder` (단계 4 사다리 연장)
- **slots**:
  1. `pain_points` — 패턴에서 도출된 페인 포인트 (Fact 층위)
  2. `needs_statements` — "OOO 하기 위해서 OOO 하고 싶다" 정형 문장. 각 문장에 `궁극적 이유(가치)` + `가치 달성을 위한 행위` 명시
  3. `needs_typing` — 각 니즈에 Explicit / Tacit / Latent 유형 태그
  4. `clustered_needs` — 유사 니즈끼리 그룹화
- **build_direction**: `bottom_up`
- **dependencies**: 2번은 1번 있을 때, 3·4번은 2번 있을 때

### ② 분기 정의
- **branch N1 · 표면 니즈에 머무름**
  - 감지: 모든 니즈가 Explicit으로만 분류
  - 대응: 빙산 모델 시각화 — "쉽게 보이는 fact 아래에 잠재된 욕구가 있어요. Tacit/Latent를 어디서 더 찾아볼 수 있을까요?"

- **branch N2 · 솔루션이 니즈 문장에 들어감**
  - 감지: "OOO 앱을 쓰고 싶다", "OOO 기능이 있으면 좋겠다" 등 솔루션·기능 명시
  - 대응: 헌법 원칙 3·4 인용. 솔루션을 떼어내 *가치·행위*로만 표현하도록 — "그 기능으로 *무엇을* 하고 싶으신 거예요?"

- **branch N3 · Fact를 Needs로 혼동**
  - 감지: 관찰된 행동을 그대로 니즈로 적음 ("앱을 3개 쓰고 있다" → "3개의 앱을 쓰고 싶다")
  - 대응: Fact는 빙산의 일각이고 그 아래 욕구가 진짜 니즈임을 짚음. 같은 사실에서 다른 니즈가 도출될 수 있음을 시연

### ③ 메서드 카탈로그
- **method `iceberg_model`** — Fact / Needs 빙산 시각화. 표면 사실 아래 잠재 욕구
- **method `needs_statement_template`** — "OOO 하기 위해서 OOO 하고 싶다" 정형. 두 OOO에 가치·행위 매핑
- **method `needs_type_taxonomy`** — Explicit(말로 표현) / Tacit(행동 관찰) / Latent(분석 도출)
- **method `pain_to_needs_fanout`** — 하나의 Pain point에서 복수 니즈 도출

### ④ 진입 점검
- **required_inputs**: 단계 4 게이트 통과
- **honest_fallback**: 패턴이 빈약하면 단계 4로 돌아가기 권유

### ⑤ 게이트 조건
- **pass_criteria**: needs_statements 최소 5개 + Tacit 또는 Latent 유형 최소 2개 (워크북 정신: 잠재 니즈 발굴이 핵심)
- **held_message**: Explicit만 있을 때 — "겉으로 드러나는 부분만 잡으셨어요. 행동 관찰에서 한 번 더 길어 올려볼까요?"

## 단계 6 · Create / Ideation

워크북 출처: Design Thinking Workbook §6 Ideation (Ground rules, SCAMPER, Bull's Eye)

### ① 산출물 스펙
- **artifact_type**: `form` (아이디어 시트 컬렉션)
- **slots** — 아이디어당 1세트:
  1. `idea_title` — 딱 맞는 제목
  2. `idea_visual` — 그림/스케치 (사용자 업로드 또는 코치 제안 후 사용자 확정)
  3. `idea_description` — 간단 설명
  4. `idea_rationale` — 어느 needs_statement를 해결하는가 + 문제점 (출처 연결 필수)
- **build_direction**: `parallel` (양 우선)
- **dependencies**: 4번은 단계 5의 needs_statements에 연결되어야 함

### ② 분기 정의
- **branch I1 · 비판 / 평가 발화**
  - 감지: 사용자가 자기 또는 가상의 다른 아이디어를 비판 ("이건 별로일 것 같은데"), 코치 자신도 동일하게 적용
  - 대응: Ground Rule 1(Defer judgement) 인용. *Ideation 단계에서는 코치도 판단 보류* — Empathize의 짚어주는 톤에서 수용·증폭 톤으로 모드 전환
  - 워크북 인용: "부정적인 발언을 하는 분과원의 얼굴에 스티커를 붙여주세요" — 이걸 코치가 농담 톤으로 환기

- **branch I2 · 양 부족**
  - 감지: 한 needs_statement당 아이디어 1-2개에 그치고 더 못 냄
  - 대응: Ground Rule 6(Go for quantity) 인용. SCAMPER 메서드 제안 — 기존 아이디어에 7가지 변형 적용

- **branch I3 · 막힘 / 발산 실패**
  - 감지: 아이디어가 전혀 안 나옴, 또는 너무 비슷한 변형만 반복
  - 대응: Bull's Eye 메서드 제안 — 동종/유사/이종 산업의 해결 방식 참조

- **branch I4 · 텍스트만 / 시각화 없음**
  - 감지: idea_visual 슬롯이 비어 있고 텍스트로만 설명
  - 대응: Ground Rule 4(Be visual) 환기. 거친 스케치라도 그리도록 — 손그림 업로드 또는 코치가 SVG 초안 생성 후 사용자 수정

### ③ 메서드 카탈로그
- **method `seven_ground_rules`** — 판단 보류 / 엉뚱한 아이디어 권장 / 살 붙이기 / 시각화 / 주제 집중 / 양 우선 / 빌려오기. Ideation 진입 시 사용자에게 함께 읽어줌
- **method `scamper`** — Substitute / Combine / Adapt / Modify·Magnify / Put to other uses / Eliminate·Minimize / Reverse·Rearrange. 변형 막혔을 때 처방
- **method `bulls_eye`** — 동종(중심)/유사(중간)/이종(바깥) 산업의 동일 니즈 해결 방식 참조. 막막함 처방
- **method `idea_sheet_5fields`** — 제목 / 작성자 / 그림 / 설명 / 이유(문제점) — 워크북 정본
- **method `build_on_others`** — 다른 아이디어에 살 붙이기 권장 (Ground Rule 3)

### ④ 진입 점검
- **required_inputs**: 단계 5 게이트 통과 (니즈 스테이트먼트 ≥5, Tacit/Latent ≥2)
- **honest_fallback**: 니즈가 표면적이면 단계 5로 돌아가기 권유 — "표면 니즈에서 시작하면 표면 아이디어가 나와요"

### ⑤ 게이트 조건
- **pass_criteria**: 아이디어 최소 10개 + 각 아이디어가 needs_statement에 연결 + idea_visual 채워진 비율 70%+
- **held_message**: 양 부족 시 SCAMPER·Bull's Eye 처방, 시각화 부족 시 Ground Rule 4 환기

## 단계 7 · Create / Idea Prioritization

워크북 출처: Design Thinking Workbook §7 Idea prioritization

### ① 산출물 스펙
- **artifact_type**: `matrix` (2x2)
- **slots**:
  1. `placement_matrix` — 가로축: 개발 난이도(Difficult ↔ Easy), 세로축: 사용자 수용성(Low ↔ High). 단계 6 아이디어 칩을 사용자가 드래그 배치
  2. `priority_curve` — 사용자/코치가 매트릭스 위에 임의 곡선을 그어 우선순위 라인 표시 (워크북 정본 방식)
  3. `ranked_list` — 좌측 우상단부터 자원·역량 고려 순위 (워크북 정본 방식)
- **build_direction**: `by_axis`
- **dependencies**: 2번은 1번에 모든 아이디어 배치된 후, 3번은 2번 곡선 그어진 후

### ② 분기 정의
- **branch P1 · 매트릭스 한쪽 쏠림**
  - 감지: 모든 아이디어가 같은 사분면(예: 수용성 高 + 난이도 高)에 몰림
  - 대응: 분포에 대한 코칭 — "쉬운 쪽에 뭘 더 넣을 수 있을까요? 작게 시작할 거리가 있을지 봐요"

- **branch P2 · 자기 평가 자신감 과잉**
  - 감지: 사용자 수용성을 사용자 *본인 판단*만으로 모두 High에 둠
  - 대응: 출처 환기 — "수용성 High로 두신 근거가 어느 응답자 인용구에서 왔는지 보여주실래요?" (단계 4 인용구 추적성 활용)

- **branch P3 · 우선순위 직관 무시**
  - 감지: 곡선 좌상단(높은 수용성·낮은 난이도) 아이디어가 있는데 우상단을 1순위로 고름
  - 대응: 워크북 정본 방식(좌측 우상단부터) 안내. 자원·역량 고려를 명시화

### ③ 메서드 카탈로그
- **method `2x2_acceptance_difficulty`** — 워크북 정본 2축. 비용은 개발 난이도에 포함
- **method `priority_curve_drawing`** — 매트릭스 위 임의 곡선으로 우선순위 라인 그리기
- **method `top_left_first`** — 좌측 우상단부터 자원·역량 따라 순위. 워크북 정본

### ④ 진입 점검
- **required_inputs**: 단계 6 게이트 통과 (아이디어 ≥10)
- **honest_fallback**: 아이디어가 모두 한 needs_statement에만 몰려 있으면 단계 6 추가 돌이키기 권유

### ⑤ 게이트 조건
- **pass_criteria**: 모든 아이디어 매트릭스 배치 + 곡선 표시 + 상위 3-5개 ranked_list 확정
- **held_message**: 분포 편향 또는 출처 부재 시 안내

## 단계 8 · Create / Concept Development

워크북 출처: Design Thinking Workbook §8 Concept Development

### ① 산출물 스펙
- **artifact_type**: `sheet` (컨셉 시트, 우선순위 상위 아이디어 그룹당 1장)
- **slots** (워크북 Concept Sheet 정본):
  1. `concept_name` — 컨셉 이름
  2. `key_features` — 주요 기능 요구사항 (단계 7 상위 아이디어들을 묶어 Feature로 발전)
  3. `concept_description` — 컨셉 설명
  4. `considerations` — 추진을 위한 자원 및 고려사항
  5. `storyboard` — 워크플로우 시나리오 (3-5컷). 이해관계자가 어떻게 체험하는지
- **build_direction**: `parallel` per concept
- **dependencies**: 5번은 1-3번 채워진 후

### ② 분기 정의
- **branch C1 · 기능 나열에 그침**
  - 감지: key_features만 채우고 description·storyboard 비어 있음
  - 대응: 워크북 정신 환기 — "컨셉 시트는 이해관계자에게 *비전·목적·핵심*을 전달하는 도구예요. 사용자가 이걸 *어떻게 체험*하는지 보여야 해요"

- **branch C2 · 스토리보드 추상화**
  - 감지: storyboard가 글로만, 상황·인물·맥락 없음
  - 대응: 워크플로우 시나리오로 구체화 — "한 응답자(단계 4 페르소나 중 1명)가 이 컨셉을 처음 만나는 순간부터 끝까지 그려볼까요?"

- **branch C3 · 단일 Feature 컨셉**
  - 감지: 1개 아이디어를 그대로 컨셉으로 가져옴, Feature 묶음 발전 없음
  - 대응: 워크북 지시 인용 — "여러 Feature를 묶어서 체험으로 발전시키는 단계예요". 단계 7 인접 아이디어와 결합 제안

### ③ 메서드 카탈로그
- **method `concept_sheet_template`** — 워크북 정본 5필드 양식
- **method `feature_grouping`** — 아이디어 그룹 → Feature 발전
- **method `storyboard_3to5frames`** — 페르소나·맥락·인터랙션·결과 4요소를 3-5컷에 담기

### ④ 진입 점검
- **required_inputs**: 단계 7 게이트 통과 (ranked_list 상위 3-5개)
- **honest_fallback**: 우선순위가 막연하면 단계 7로 돌아가기

### ⑤ 게이트 조건
- **pass_criteria**: 컨셉 시트 최소 1장 완성 + 스토리보드 채워짐
- **held_message**: 워크북 범위 내 마지막 단계임을 알림. 다음 단계(프로토타입)는 서비스 확장 영역임을 명시

## 워크북 범위와 서비스 확장의 경계

단계 1-8은 HiDD 워크북 정본 범위. 이후 단계는 본 서비스가 확장하는 영역으로, 단계 8의 Concept Sheet가 다음 단계의 입력이 됨:

- **단계 9 · 프로토타입** — Concept Sheet + Storyboard → 화면 설계 JSON → 화면별 코드 생성 → iframe 렌더링
- **단계 10 · 검증** — 단계 2-3의 디자인 리서치 메서드(특히 In-context Interview)를 *사용성 테스트*로 재사용. 워크북이 한 번이 아니라 여러 번 돈다는 점이 자연스러운 회귀 루프
- **단계 11 · 사업화** — 비즈니스 모델 확정, GTM, 다음 액션 가이드

UI에서 단계 1-8과 9-11은 시각적으로 구분(예: 워크북 마크 vs 확장 마크)하여 사용자가 *어디까지가 검증된 방법론이고 어디부터가 서비스 고유 영역인지* 알 수 있게 함.

## 새 단계 추가 시 작성 체크리스트

1. artifact_type 4종(form/ladder/matrix/sheet) 중 선택
2. slots 정의 + dependencies 명시
3. 단계 고유 분기 3-5개 (감지 + 대응 + 헌법 인용)

# 디자인 무드 가이드 v1

> 디자인씽킹 코칭 서비스의 시각 언어 가이드. Predic AI의 절제된 SaaS 무드에 HiDD 로고의 색상 정체성을 결합한다.
> 이 문서는 Claude로 화면을 디자인할 때 옆에 두고 참조하는 도구이며, 새 화면을 만들 때마다 *시맨틱 결정 → 색 적용 → 컴포넌트 선택* 순서로 사용한다.

---

## 15초 핵심 원칙

새 화면을 만들기 전에 이 다섯 원칙만 머리에 두면 무드의 80%가 맞춰진다.

1. **노랑은 통찰의 순간에만**. 코치 제안 도착, 시각 재정의, Why 사다리 핵심 단, 발견 배지, 완료 체크. 모든 곳에 노랑을 쓰면 의미가 사라진다.
2. **차콜은 결정과 액션의 무게**. 메인 텍스트, 주요 버튼, 헤더. 화면에서 가장 무거운 색.
3. **회백색은 양보하는 색**. 페이지 배경, 카드 면, 보조 칩. 산출물이 시선의 주인공이 되도록 모든 chrome이 물러난다.
4. **한 화면에 통찰의 자리는 한 곳**. 노랑이 켜진 자리가 한 곳이면 의미가 가장 무거워진다. 두 곳 이상이면 모두 같은 통찰의 결로 묶여야 한다.
5. **노랑 없는 화면은 의도된 결정**. 진입점 선택 같은 *통찰 이전의 자리*는 회백색·차콜만으로 차분하게 흐른다.

### 사용 규칙

**노랑을 쓰는 자리** (한 화면에 한두 곳만)
- 코치 제안 칩 — 새로운 시각 도착
- 시각 재정의 적용 칩 — 분기 작동
- Why 사다리 핵심 단 — 깊은 동기
- 가설 신뢰도 발견 배지
- 단계·슬롯 완료 체크
- 카드 헤더 배지 (확정 · 가설, 코치 제안 등)
- 진행률 바 (슬롯 완성 시)
- 코치 아바타 배경 (제안 중 · 짚어주는 중 · 격려 중 모드)

**노랑을 안 쓰는 자리**
- 일반 진행 표시, 일반 버튼
- 텍스트 본문, 보조 라벨
- 카드 테두리, 분할선
- 비활성 상태
- 단순 알림, 정보 표시
- 듣는 중 · 묻는 중 코치 모드

## 2. 타이포그래피

### 사이즈 시스템

본문 위주의 디자인이므로 *큰 헤드라인이 거의 없다*. 카드 안의 정보 위계는 *색과 weight*로 만들지 *크기*로 만들지 않는다.

| 토큰 | 크기 | 행간 | 용도 |
|---|---|---|---|
| `--text-page-title` | 18px | 1.4 | 페이지 제목 (드물게 사용) |
| `--text-card-title` | 13px | 1.4 | 카드 헤더 |
| `--text-body` | 13px | 1.65 | 코치 발화 본문 |
| `--text-card-body` | 12.5px | 1.5 | 카드 내부 본문 |
| `--text-meta` | 11.5px | 1.45 | 메타 정보, 라벨 |
| `--text-helper` | 11px | 1.5 | 보조 안내, 부제 |
| `--text-hint` | 10.5px | 1.4 | 힌트, 메타 칩 라벨 |

### Weight 시스템

| Weight | 용도 |
|---|---|
| `400` (regular) | 모든 본문, 보조 텍스트 |
| `500` (medium) | 핵심 단어 강조, 카드 헤더, 액션 버튼 |

**Bold(700)는 쓰지 않는다.** Medium 한 단계만으로 강조한다. 무드의 절제된 톤이 깨지지 않게.

### 강조 패턴

본문 안에서 핵심 단어를 강조할 때 두 가지 패턴을 쓴다.

```html
<span style="font-weight: 500;">핵심 단어</span>
```
일반 본문 안에서 한두 단어만 weight로 강조. 색은 그대로 차콜.

```html
<span style="color: var(--color-muted-gray);">부연 설명</span>
```
주된 메시지 안에서 살짝 양보하는 톤. 같은 발화 안에서도 위계를 만든다.

이탤릭은 *인용·고유 표현*에만 쓴다 — `*사용자 발화*`, `*워크북 용어*` 같은 자리.

## 3. Spacing & Radius

### Spacing 8px grid

기본 단위 8px. 미세 조정은 4px 단위로.

| 토큰 | 값 | 용도 |
|---|---|---|
| `--space-1` | `4px` | 인접 라벨과 본문 사이 |
| `--space-2` | `7px` | 칩 내부 패딩 |
| `--space-3` | `11px` | 카드 내부 발화 간격 |
| `--space-4` | `14px` | 카드 내부 패딩 |
| `--space-5` | `16px` | 보드 내부 패딩 |
| `--space-6` | `22px` | 페이지 외곽 패딩 |

### Border Radius

라운드는 *큰 편*이다. Predic AI 무드의 부드러운 SaaS 톤.

| 토큰 | 값 | 용도 |
|---|---|---|
| `--radius-chip` | `6px` | 작은 칩, 라벨 배지 |
| `--radius-card-inner` | `8px` | 카드 안 후보 박스, 내부 카드 |
| `--radius-button` | `10px` | 버튼, 입력 필드 |
| `--radius-card` | `12px` | 카드 (Why 사다리 핵심 단 등) |
| `--radius-card-large` | `14px` | 메인 카드 (분할 화면 양쪽) |
| `--radius-page` | `16px` | 페이지 컨테이너 |

### Border Width

| 토큰 | 값 | 용도 |
|---|---|---|
| `--border-thin` | `1px` | 기본 카드 테두리 |
| `--border-emphasis` | `1.5px` | 코치 제안 상태, Why 사다리 핵심 단, 강조 카드 |

**2px 이상은 쓰지 않는다.** 1px와 1.5px 두 단계만으로 위계를 만든다.

## 4. 시맨틱 → 시각 매핑

이게 가이드의 가장 중요한 부분. *어떤 상태에 어떤 시각이 가는지*를 명시한다. 새 화면을 그릴 때 이 표를 먼저 보고 시각을 정한다.

### 슬롯 상태 4종

| 상태 | 테두리 | 배경 | 텍스트 | 라벨 위치 |
|---|---|---|---|---|
| `미작성` | `1px dashed #E5E4DD` | 없음 | `#A09E94` | 우측 — "아직" / "나중에" |
| `작성 중` | `1.5px solid #2D2D2A` | `#FFFFFF` | `#2D2D2A` | 우측 — "지금 작성" |
| `코치 제안` | `1.5px solid #F4C430` | `#FFFDF4` | `#2D2D2A` | 우측 — 노랑 배지 |
| `확정` | `1px solid #E5E4DD` | `#FFFFFF` | `#2D2D2A` | 우측 — 노랑 체크 + 항목 수 |

**작성 중 상태가 노랑이 아닌 차콜 테두리**인 게 중요한 결정. 사용자가 *작성 중*은 *통찰이 아니라 결정의 자리*다. 노랑은 *코치 제안*에만 켜진다.

확정 상태가 *흰 배경 + 회백색 테두리*로 차분하게 가라앉는 것도 결정. 진행되어 *시선의 주인공에서 양보*하는 시각 언어.

### 카드 접힘 상태

확정된 카드는 화면 공간 절약을 위해 한 줄로 접힌다. 접힘 정도가 *진행 깊이*에 따라 단계적으로 짙어진다.

| 시점 | 불투명도 | 용도 |
|---|---|---|
| 직전 카드 작업 중 | `0.78` | 방금 끝난 카드 |
| 두 카드 더 작업 후 | `0.72` | 더 가라앉음 |
| 슬롯 완성 시점 | `0.62-0.68` | 거의 잔향만 |

### 코치 모드 5종 → 아바타

| 모드 | 아바타 배경 | 아이콘 | 의미 |
|---|---|---|---|
| `듣는 중` | `#FFFFFF` | `ti-ear` 차콜 | 사용자 입력 대기 |
| `묻는 중` | `#FFFFFF` | `ti-help` 차콜 | 소크라테스식 질문 |
| `제안 중` | `#F4C430` | `ti-sparkles` 차콜 | 코치 후보 도착 |
| `짚어주는 중` | `#F4C430` | `ti-bulb` 차콜 | 분기 작동, 시각 재정의 |
| `격려 중` | `#F4C430` | `ti-mood-smile` 차콜 | 슬롯 완성, 따뜻한 호응 |

**노랑 아바타 3종은 모두 같은 #F4C430**. 아이콘만 다르다. *모드의 미세한 차이는 아이콘으로, 통찰의 결은 노랑으로 통일*하는 결정.

흰 아바타 2종(듣는 중·묻는 중)은 *통찰 이전의 자리*. 코치가 능동적으로 통찰을 만들지 않는 호흡.

### 신뢰도 배지

| 신뢰도 | 색 토큰 | 시각 |
|---|---|---|
| `발견` | 노랑 시스템 | `#FCF3D6 배경 + #8B6914 텍스트` |
| `가설` | 차콜 / 회백색 | `#FCF3D6 배경 + #8B6914 텍스트` (단, 본문에 "가설" 명시) |

발견과 가설 모두 *노랑 톤*이다. 두 가지 모두 *통찰의 결*이기 때문. *발견*은 인터뷰 증거로 검증된 통찰, *가설*은 검증 전 통찰. 두 시점이 같은 색 가족으로 묶인다.

### 출처 추적 칩 (source_refs)

```html
<div style="background: #FFFFFF; border: 1px solid #CEE0F5; ...">
  <i class="ti ti-link"></i>
  N번에서 자라남 · 인용 텍스트
</div>
```

이전 카드 항목이 현재 카드에 출처로 연결될 때. *연한 파란 톤 테두리*(`#CEE0F5`)를 예외적으로 사용 — 출처는 *통찰이 아닌 사실*이므로 노랑이 아니고, 회백색만으로는 시각 가중치가 부족하기 때문. **유일하게 노랑·차콜·회백 시스템 밖의 색을 쓰는 자리.**

### 시각 재정의 칩 (분기 작동)

```html
<div style="background: #FCF3D6; border-radius: 8px; padding: 9px 11px;">
  <i class="ti ti-bulb" style="color: #8B6914;"></i>
  <strong>시각 재정의 적용</strong>
  <div style="color: #6B6A66;">전 시각 → 후 시각</div>
</div>
```

분기 작동의 영구 기록. 카드 안에 *통찰이 일어난 자리*로 보존된다. 사용자가 며칠 후 카드를 다시 봐도 *여기서 코치가 시각을 재정의했구나*가 기억의 갈고리로 남음.

## 5. 컴포넌트 패턴

자주 등장하는 컴포넌트의 마크업과 시각 패턴.

### 페이지 컨테이너

```html
<div style="background: #FAFAF7; border-radius: 16px; padding: 16px;">
  <!-- 분할 화면이면 grid-template-columns: 1fr 1fr; gap: 14px; -->
  <!-- 단일 컬럼이면 max-width: 480px; margin: 0 auto; -->
</div>
```

### 메인 카드 (대화 패널 / 산출물 보드)

```html
<div style="background: #FFFFFF; border: 1px solid #E5E4DD;
            border-radius: 14px; padding: 16px;">
  <!-- 카드 내용 -->
</div>
```

### 코치 아바타 헤더

```html
<div style="display: flex; align-items: center; gap: 10px;
            padding-bottom: 11px; border-bottom: 1px solid #F0EFE9;">
  <div style="width: 38px; height: 38px; border-radius: 50%;
              background: #F4C430;
              /* 또는 #FFFFFF + border: 1px solid #E5E4DD; */
              display: flex; align-items: center; justify-content: center;">
    <i class="ti ti-bulb" style="font-size: 20px; color: #2D2D2A;"></i>
  </div>
  <div>
    <div style="font-size: 13px; font-weight: 500; color: #2D2D2A;">Kevin</div>
    <div style="font-size: 11px; color: #6B6A66;">짚어주는 중 · 시각 재정의</div>
  </div>
</div>
```

라벨에 *모드 + 한 단어 메타*를 결합 — *짚어주는 중 · 시각 재정의*, *묻는 중 · 카드 전환* 같은 식.

### 코치 발화 버블

```html
<!-- 주된 메시지 -->
<div style="font-size: 13px; line-height: 1.7; color: #2D2D2A;
            background: #F8F7F1; border-radius: 12px; padding: 11px 13px;">
  발화 본문 <span style="font-weight: 500;">핵심 단어</span>
</div>

<!-- 부연 메시지 -->
<div style="font-size: 13px; line-height: 1.7; color: #6B6A66;
            background: #F8F7F1; border-radius: 12px; padding: 11px 13px;">
  주된 메시지를 보충하는 흐린 톤
</div>
```

### 사용자 발화 버블

```html
<div style="align-self: flex-end;
            font-size: 13px; line-height: 1.6; color: #2D2D2A;
            background: #F3F2EC; border-radius: 12px;
            padding: 10px 13px; max-width: 86%;">
  사용자 입력 텍스트
</div>
```

### 입력 필드 (비활성 상태)

```html
<div style="border: 1px solid #E5E4DD; border-radius: 10px;
            padding: 10px 13px; font-size: 12.5px; color: #A09E94;
            background: #FFFFFF;
            display: flex; align-items: center; justify-content: space-between;">
  <span>답장 입력…</span>
  <i class="ti ti-arrow-up"></i>
</div>
```

### 액션 버튼 두 종류

```html
<!-- 주된 액션 (차콜) -->
<div style="background: #2D2D2A; color: #FFFFFF;
            font-size: 11px; font-weight: 500; text-align: center;
            padding: 8px 10px; border-radius: 8px; cursor: pointer;">
  채택 · 자세히 적기
</div>

<!-- 보조 액션 (흰 + 테두리) -->
<div style="background: #FFFFFF; color: #2D2D2A;
            border: 1px solid #E5E4DD;
            font-size: 11px; text-align: center;
            padding: 8px 10px; border-radius: 8px; cursor: pointer;">
  다듬기
</div>
```

### 진행률 바

```html
<div style="height: 3px; background: #F0EFE9; border-radius: 2px;">
  <!-- 진행 중: 차콜 -->
  <div style="width: 60%; height: 100%; background: #2D2D2A; border-radius: 2px;"></div>

  <!-- 완성 시: 노랑 -->
  <div style="width: 100%; height: 100%; background: #F4C430; border-radius: 2px;"></div>
</div>
```

### 카드 메타 칩 (출발점, 안내)

```html
<div style="background: #FAFAF7; border: 1px solid #E5E4DD;
            border-radius: 10px; padding: 8px 12px;
            display: flex; align-items: center; gap: 7px;">
  <i class="ti ti-target" style="font-size: 13px; color: #6B6A66;"></i>
  <div style="font-size: 11.5px; color: #2D2D2A;">
    출발점 · <span style="color: #6B6A66; font-style: italic;">맥락 텍스트</span>
  </div>
</div>
```

### 통찰 칩 (시각 재정의, 가설 신뢰도 등)

```html
<div style="background: #FCF3D6; border-radius: 8px;
```

## 6. 새 화면을 그릴 때 워크플로우

이 가이드를 *읽는 문서*가 아니라 *쓰는 도구*로 활용하는 5단계.

### 1단계 · 시맨틱 결정

화면에 들어갈 *시맨틱 요소*를 먼저 정한다.
- 이 화면에 *통찰*이 일어나는가? 어디서? (노랑 자리)
- 이 화면에 *결정*이 일어나는가? 어디서? (차콜 자리)
- 슬롯/카드의 상태는 무엇인가? (미작성·작성중·코치제안·확정)
- 코치는 어떤 모드인가? (5종 중 하나)

### 2단계 · 통찰 자리 한 곳 결정

화면 전체에서 *노랑이 가장 진하게 켜지는 한 자리*를 정한다. 그 자리가 화면의 시각 중심이 된다. 분기 V4 컷의 시각 재정의 칩, Why 사다리의 3단 노랑 마커, 진입점 화면의 *노랑 없음*.

### 3단계 · 시맨틱 → 시각 매핑 표 참조

§4의 매핑 표를 보고 각 요소의 색·테두리·배경을 정한다. 임의로 색을 선택하지 않는다.

### 4단계 · 컴포넌트 패턴 조립

§5의 마크업 패턴을 복사해서 시작하고, 콘텐츠만 갈아끼운다. 새 컴포넌트는 패턴이 명백히 부족할 때만 만든다.

### 5단계 · 무드 검수

15초 핵심 원칙 5개를 다시 본다.
- 노랑이 한 자리에만 켜졌나? (또는 같은 통찰의 결로 묶였나?)
- 차콜이 너무 많지 않나?
- 회백색이 산출물을 양보하고 있나?
- 임의의 색이 들어가 있지 않나?

## 7. 자주 하는 실수

새 화면을 만들 때 자주 빠지는 함정들. Claude로 디자인할 때 특히 주의.

### 실수 1 · 노랑을 일반 강조로 씀

가장 흔한 실수. 카드 헤더 강조, 호버 상태, 일반 버튼에 노랑을 쓰면 *통찰의 의미*가 사라진다. 노랑은 *시맨틱 색*이지 *디자인 색*이 아니다.

**해결**: 의심스러우면 안 쓴다. 차콜 한 톤이 답인 경우가 90%.

### 실수 2 · 시스템 변수를 그대로 사용

Claude 기본 디자인 시스템에는 `--color-text-info`(파란 톤), `--color-text-success`(초록), `--color-text-warning`(주황) 같은 변수가 있다. **이 변수를 그대로 쓰면 무드가 깨진다.** 우리는 *모든 의미 색을 노랑/차콜/회백 시스템으로 통합*했다.

**해결**: 시스템 변수 대신 §1의 색 토큰을 인라인 hex로 사용한다.

### 실수 3 · 라운드를 너무 작게

`border-radius: 4px` 같은 작은 라운드는 무드와 안 맞는다. Predic AI 톤은 *부드러운 SaaS*라 라운드가 큰 편이다.

**해결**: 최소 6px(칩), 보통 10-12px(카드 내부 박스), 큰 카드는 14px.

### 실수 4 · Bold 사용

Bold(700)는 *시각적 무게가 너무 무겁다*. 우리 무드는 *절제된 강조*가 핵심이라 medium(500)으로 충분하다.

**해결**: `font-weight: 500;`만 쓰고 `bold`는 쓰지 않는다.

### 실수 5 · 카드 안에 카드 안에 카드

3단 중첩 이상의 카드 안 카드는 시각이 복잡해진다. 우리 무드는 *최대 2단 중첩*까지만 — 메인 카드 + 내부 박스. 더 깊은 중첩이 필요해 보이면 *시맨틱이 잘못된 것*이다.

**해결**: 3단이 필요해 보이면 두 카드로 분리하거나, 중첩 대신 *섹션 구분*으로 표현.

### 실수 6 · 차콜 텍스트를 노랑 위에 사용

`#FCF3D6 노랑 틴트` 위에 `#2D2D2A 차콜 텍스트`는 대비가 충분하지만, `#F4C430 진한 노랑` 위에는 `#2D2D2A`가 약간 거칠다.

**해결**: 진한 노랑 위 텍스트는 `#8B6914 짙은 노랑`을 쓰거나, 진한 노랑은 *배경이 아닌 액센트*(테두리·작은 점)로만 쓴다.

## 8. Claude로 디자인할 때 프롬프트 패턴

이 가이드를 Claude에게 전달할 때 효과적인 패턴.

### 시작 패턴

```
[디자인 무드 가이드 v1을 첨부]

이 무드 가이드를 따라 [화면 설명]을 만들어주세요.

먼저 §6 워크플로우의 1-2단계를 따라:
- 이 화면의 시맨틱 요소
- 통찰 자리 한 곳
을 먼저 정리해 보여주세요. 시각 결정은 그 다음에.
```

이렇게 *시맨틱부터 정리하라*고 요청하면 색·레이아웃을 임의로 정하지 않고 가이드를 따른다.

### 검수 패턴

```
완성된 화면을 §7 자주 하는 실수 6가지로 검수해주세요.
- 노랑이 일반 강조로 쓰이지 않았나
- 시스템 변수가 그대로 들어가지 않았나
- 라운드가 너무 작지 않나
- Bold가 쓰이지 않았나
- 3단 중첩이 없나
- 노랑 위 텍스트 대비가 적절한가
```

### 무드 검증 패턴

```
15초 핵심 원칙 5개를 다시 보고 이 화면을 검토해주세요.
어긋난 자리가 있다면 어떻게 고쳐야 할지 알려주세요.
```

## 9. 무드의 정체성 한 줄

기억해야 할 한 줄이 있다면 이것이다.

> **회백색 무드의 차분함 안에서, 통찰이 일어나는 한 자리에 노랑이 켜진다.**

이 한 줄이 모든 결정의 기준이다. 의심스러운 디자인 결정을 내릴 때 이 문장으로 돌아오면 답이 보인다. 노랑이 한 자리에 켜져 있는가? 차분함이 유지되는가? 통찰이 일어나는 자리인가? 셋 다 *예*면 맞는 결정이고, 하나라도 *아니오*면 다시 봐야 한다.

HiDD 로고의 *"!Insight" 노랑*이 우리 서비스 전체의 시각 정체성이다. 모든 작은 결정이 그 한 가지 메시지를 강화하도록 묶인다.
